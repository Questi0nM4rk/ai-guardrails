# Design: AI Guardrails v3.1 — Final Release

## Problem

v3.0 proved the three-ring defense model works (agent hooks → commit hooks → CI gate). But adoption friction remains: init is monolithic, profiles are cosmetic, inline suppression isn't honored by check, distribution is binary-only, and .NET is a stub. Teams need a 30-second setup, real profile enforcement, and a way to permanently allow specific issues without baseline hacks.

### Success Metrics

| Metric | Target | How Measured |
|--------|--------|--------------|
| Init time (interactive) | < 30 seconds | Timer on standard machine |
| Init time (--yes) | < 10 seconds | CI measurement |
| Test count | 1,036+ (896 + ~140 new) | `bun test` |
| Backward compat | --yes identical to v3.0 | Golden snapshot test |
| Platform coverage | 4 binaries + npm | Release artifacts |

### Constraints

- **Hard**: Bun >= 1.2.0, strict TS, no `as` casts, no TUI framework (readline only), max 200 LOC/file
- **Soft**: Prefer composition, fakes over mocks, Gherkin for behavior tests
- **Assumptions**: Bun cross-compilation stable, ruff/biome maintained, Commander.js completion API exists

### Scope

- **In**: Interactive init, profile filtering, allow integration, shell completion, .NET runner, npm publish
- **Out**: Triage CLI, burn-down, hook audit, monorepo, cloud/SaaS

## Solution

### Chosen Approach

Six independent features built on the existing composable architecture. Each plugs into established extension points (LanguagePlugin, LinterRunner, ConfigGenerator, Pipeline). The init wizard is the only structural change — everything else is additive.

### Alternatives Considered

| Option | Pros | Cons | Why Not |
|--------|------|------|---------|
| TUI framework (ink/blessed) | Rich interactive UI | Adds dependency, breaks readline-only constraint | Hard constraint violation |
| Monolithic init rewrite | Simpler, fewer files | No modularity, can't add features without touching one file | Doesn't scale |
| Profile via separate config files | Clear separation | File proliferation, merge complexity | One config file is simpler |

### Trade-offs Accepted

| Chose | Over | Rationale |
|-------|------|-----------|
| Readline prompts | Rich TUI | Constraint: no TUI deps. Readline works everywhere. |
| npm Linux-only binary | Platform-specific npm packages | v3.1 simplicity. v3.2 adds `optionalDependencies` per platform. |
| Regex allow-comment parsing | Full AST parsing | Comments are simple patterns. AST parsing is overkill for `// ai-guardrails-allow`. |
| Built-in profiles only | Custom user profiles | v3.1 scope. Custom profiles deferred to v4. |

## Integration Points

- **Init → Config → Generators**: Profile selected in init → stored in config.toml → generators read `config.profile` and adjust rules
- **Allow → Check → Baseline**: Allow comments filter BEFORE baseline classification. Snapshot excludes allowed issues.
- **Init → Per-file conflict**: Each module checks if target file exists + has hash header before writing. Prompts merge/skip.
- **.NET → Plugin → Check**: dotnet.ts returns dotnet-build runner. Check pipeline runs it like any other runner.
- **npm → Release**: Same binary, published to npm registry after GitHub Release.

## Testing Strategy

~140 new tests across init modules (26), dependencies (20), wizard (10), CLI flags (15), allow parsing (30), .NET parser (20), backward compat (5), error handling (10), completion (4).

Target: 1,036+ total tests, 90%+ coverage on new code.

## Evolution

- **Stable for**: Current 6-feature scope with 3 built-in profiles
- **Revisit if**: Module count > 20 (add sub-categories), custom profiles demanded (add config schema), allow comments unreliable (add validation warnings)
- **Deferred**: Triage CLI (v3.2), burn-down reporting (v3.2), hook audit (v3.2), monorepo (v4)
