# SPEC-005: Hooks

## Status: Draft

---

## Overview

Two hook categories:

1. **Claude Code PreToolUse hooks** — intercept tool calls in-process before
   execution. Installed globally into `~/.claude/settings.json` by `ai-guardrails install`.
2. **Lefthook commit hooks** — run at pre-commit time. Installed per-project by
   `ai-guardrails init` via `lefthook.yml` generation.

All hooks are implemented as subcommands of the main binary:
`ai-guardrails hook <name>`. No separate Python scripts, no shell script wrappers.
The binary IS the hook runner.

## Claude Code PreToolUse Hooks

Installed into `~/.claude/settings.json`:

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          { "type": "command", "command": "ai-guardrails hook dangerous-cmd" },
          { "type": "command", "command": "ai-guardrails hook protect-configs" }
        ]
      }
    ]
  }
}
```

### TypeScript hook implementation pattern

Claude Code does not natively support TypeScript hooks (issue #4786 open as of March 2026).
The two supported patterns:

**Pattern A (recommended for this project):** `bun run hook.ts`
- Startup: 8–15ms (vs 200–400ms for `npx tsx`)
- Requires Bun on the machine — acceptable since `ai-guardrails install` is a dev-machine setup
- Used in `~/.claude/settings.json` after `ai-guardrails install`

```json
{ "type": "command", "command": "ai-guardrails hook dangerous-cmd" }
```
The binary invokes the hook subcommand directly — no separate script needed.

**Pattern B (for hooks generated into target projects):** `node script.js`
- Cannot assume Bun in target projects
- Hooks generated by `ai-guardrails init` for target project use pre-compiled JS or shell

### stdin/stdout contract

```typescript
// hooks/types.ts — shared across all hook implementations

export interface HookInput {
  session_id: string;
  transcript_path: string;
  cwd: string;
  hook_event_name: "PreToolUse" | "PostToolUse" | "Notification" | "Stop" | string;
  tool_name: string;       // "Bash", "Write", "Edit", etc.
  tool_input: Record<string, unknown>;  // shape varies per tool_name
}

export interface HookOutput {
  hookSpecificOutput: {
    hookEventName: "PreToolUse";
    permissionDecision: "allow" | "deny" | "ask";
    permissionDecisionReason?: string;
  };
}

// Bash tool_input shape:
export interface BashToolInput {
  command: string;
  description?: string;
  restart?: boolean;
}

// Write/Edit tool_input shape:
export interface WriteToolInput {
  file_path: string;
  content?: string;
  old_str?: string;
  new_str?: string;
}
```

**Exit codes:**
- `0` — allow / success (stdout JSON parsed if present)
- `2` — blocking deny (stderr message sent to Claude as error)
- other non-zero — non-blocking warning (logged, action proceeds)

### Hook runner utility (`hooks/runner.ts`)

```typescript
import type { HookInput, HookOutput } from "./types.ts";

export async function readHookInput(): Promise<HookInput> {
  const chunks: Buffer[] = [];
  for await (const chunk of process.stdin) chunks.push(chunk);
  return JSON.parse(Buffer.concat(chunks).toString("utf8")) as HookInput;
}

export function allow(): never {
  process.stdout.write(JSON.stringify({
    hookSpecificOutput: { hookEventName: "PreToolUse", permissionDecision: "allow" }
  }));
  process.exit(0);
}

export function deny(reason: string): never {
  process.stderr.write(reason + "\n");
  process.exit(2);
}
```

### Available hook events (10 total)

`PreToolUse`, `PostToolUse`, `Notification`, `UserPromptSubmit`, `Stop`,
`SubagentStop`, `PreCompact`, `SessionStart`, `SessionEnd`, `PermissionRequest`

### everything-claude pattern (reference implementation)

The `affaan-m/everything-claude-code` repo (canonical hackathon winner) uses plain
Node.js scripts with a `run-with-flags.js` dispatcher. All hooks are `.js` not `.ts`.
The `CLAUDE_PLUGIN_ROOT` env var provides portability. Hooks use `async: true` with
timeout for non-blocking background work. This confirms: use `bun run` for our hooks
(faster startup), `node` for generated target-project hooks.

### `hook dangerous-cmd`

**Input:** Claude Code passes the Bash tool input via stdin as JSON:
```json
{ "command": "rm -rf /tmp/build", "description": "..." }
```

**Logic:**
```typescript
const BLOCKED_PATTERNS = [
  /rm\s+-rf?\s+[^/\s]*\/?\s*$|rm\s+-rf\s+\//,  // rm -rf /path or rm -rf /
  /git\s+push\s+.*--force(?!-with-lease)/,        // force push (not --force-with-lease)
  /git\s+reset\s+--hard/,                          // hard reset
  /git\s+checkout\s+--/,                           // discard working tree
  /git\s+clean\s+-[a-z]*f/,                        // force clean
  /git\s+commit\s+.*--no-verify/,                  // skip hooks
];

// Returns exit 0 (allow) or exit 1 + message (block)
```

**False positive handling:** The current ruff.toml `rm -rf /tmp/...` false positive
from the old broad pattern is fixed with the more precise regex above. The pattern
`/path/to/dir` is allowed; only bare `rm -rf /` or trailing-slash patterns block.

### `hook protect-configs`

**Logic:** If the Bash command would write to any managed config file, block it.


## Lefthook Commit Hooks

Generated into `lefthook.yml` by `generators/lefthook.ts`. Structure:

```yaml
pre-commit:
  commands:
    format-and-stage:
      run: ai-guardrails hook format-stage
      stage_fixed: true    # re-stage auto-fixed files
      fail_text: "Format failed — check output above"

    check-suppress-comments:
      glob: "*.{py,ts,js,tsx,jsx,rs,go,cs,lua,sh}"
      run: ai-guardrails hook suppress-comments {staged_files}
      fail_text: "Suppression comments detected — use ai-guardrails-allow instead"

    ruff-lint:
      glob: "*.py"
      run: ruff check {staged_files}
      fail_text: "Ruff lint errors found"

    mypy:
      glob: "*.py"
      run: mypy --no-error-summary {staged_files}
      fail_text: "mypy type errors found"

    gitleaks:
      run: gitleaks protect --staged --no-banner
      fail_text: "Potential secrets detected"

    semgrep:
      glob: "*.{py,ts,js,go,rs}"
      run: semgrep --config=auto --quiet {staged_files}
      fail_text: "Semgrep security findings"

    no-commits-to-main:
      run: |
        branch=$(git rev-parse --abbrev-ref HEAD)
        if [ "$branch" = "main" ] || [ "$branch" = "master" ]; then
          echo "Direct commits to main are not allowed"
          exit 1
        fi
      fail_text: "Do not commit directly to main"
```

### `hook format-stage`

Runs formatters on staged files and re-stages the results. Lefthook's
`stage_fixed: true` handles the re-staging.

```typescript
// For each detected language, run its formatter:
// Python:  ruff format <staged_py_files>
// TS/JS:   biome format --write <staged_ts_files>
// Rust:    rustfmt <staged_rs_files>
// Go:      gofmt -w <staged_go_files>
// Lua:     stylua <staged_lua_files>
// C/C++:   clang-format -i <staged_cpp_files>
```

### `hook suppress-comments`

Scans staged files for any suppression comment pattern across all languages.
Any match fails the commit with a message pointing to the `ai-guardrails-allow` syntax.

```typescript
const SUPPRESSION_PATTERNS: Record<string, RegExp[]> = {
  python: [
    /# noqa/,
    /# type:\s*ignore/,
    /# pragma:\s*no cover/,
    /# pylint:\s*disable/,
  ],
  typescript: [
    /\/\/\s*@ts-ignore/,
    /\/\/\s*@ts-nocheck/,
    /eslint-disable/,
    /\/\*\s*tslint:disable/,
  ],
  rust: [
    /#\[allow\(/,
    /#!\[allow\(/,
  ],
  go: [
    /\/\/nolint/,
    /\/\/\s*nolint/,
  ],
  csharp: [
    /#pragma warning disable/,
    /\[SuppressMessage/,
  ],
  lua: [
    /--\s*luacheck:\s*ignore/,
    /--\s*luacheck:\s*disable/,
  ],
  shell: [
    /# shellcheck disable/,
  ],
  cpp: [
    /\/\/ NOLINT/,
    /#pragma diagnostic ignored/,
    /#pragma GCC diagnostic ignored/,
  ],
};

// Exception: lines also containing "ai-guardrails-allow" are NOT flagged.
// The ai-guardrails-allow comment IS the permitted suppression mechanism.
```

## Hook Installation (`generators/lefthook.ts`)

The generated `lefthook.yml` is language-aware:
- Python sections only included if Python detected
- TS sections only if TypeScript detected
- etc.

Language detection for hook generation runs at `init` time and is stored in
`.ai-guardrails/config.toml` as the `detected_languages` list, so `generate`
doesn't need to re-detect.

```toml
# .ai-guardrails/config.toml
detected_languages = ["python", "shell"]  # written by init, used by generate
```

## Hash Header Verification

Every generated config file has a hash header as the first line:

```
# ai-guardrails:sha256=a3f4b2c1...
```

The `protect-configs` hook checks this header exists before blocking edits.
The `generate --check` command verifies the hash matches. The hash covers
all content below the header line.

```typescript
// utils/hash.ts
export const HASH_PREFIX = "# ai-guardrails:sha256=";

export function makeHashHeader(content: string): string {
  const hash = createHash("sha256").update(content).digest("hex");
  return `${HASH_PREFIX}${hash}`;
}

export function verifyHash(fileContent: string): boolean {
  const [headerLine, ...rest] = fileContent.split("\n");
  if (!headerLine.startsWith(HASH_PREFIX)) return false;
  const storedHash = headerLine.slice(HASH_PREFIX.length).trim();
  const body = rest.join("\n");
  const computed = createHash("sha256").update(body).digest("hex");
  return storedHash === computed;
}
```

Files without a hash header are considered unmanaged — the hook does not
block edits to them.
